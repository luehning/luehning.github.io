Agile 101 Scrum 和 XP的那些事

发表于 2014-03-01   |   分类于 Software project   |   | 阅读量 次

什么是Agile

  Agile 又称敏捷开发，是一种从1990年代开始逐渐引起广泛关注的新型软件开发方法，是一种应对快速变化的需求的一种软件开发能力。特点：以人为核心、迭代、循序渐进。

Agile 十二条军规

  无规则不成方圆，Agile开发的魅力就在于它的面向客户，快速响应的价值观，为了更好的执行，规定了下面的12条军规。

  1. 满足客户 The Highest Priority is Customer Satisfaction.
  2. 欢迎对需求提出变更 Welcome Changing Requirements
  3. Deliver Working Software Frequently
  4. Work Together Daily
  5. Build Projects Around Motivated Individuals
  6. Face-to-Face Conversation is Best for Conveying Information
  7. Working Software is the Primary Measure of Progress
  8. Sustainable Pace is Key
  9. Don’t Get Rusty
  10. Simplicity is Essential
  11. Self-Organizing Teams Are Best
  12. Empower Teams to Inspect and Adapt Behavior

Agile种类

  1. scrum
  2. eXtreme Programing(XP)
  3. adaptive software development(ASD)
  4. feature driven development(FDD)
  5. dynamic systems development method(DSDM)
  6. crystal family
  7. lean
  8. Agile unified process (UP,AUP)

Scrum

  Scrum的本来的意思是“争球”，是体育运动的一个专业术语。

Scrum开发流程中的三大角色

  1. Product Owner 即产品负责人，负责确定产品的功能和达到要求的标准，指定软件的发布日期和交付的内容，同时有权力接受或拒绝开发团队的工作成果。
  2. Scrum Master 即流程管理员，负责整个Scrum流程在项目中的顺利实施和进行，以及清除挡在客户和开发工作之间的沟通障碍，使得客户可以直接驱动开发。
  3. Scrum Team 即开发团队，负责软件产品在Scrum规定流程下进行开发工作，人数控制在5~10人左右，每个成员可能负责不同的技术方面，但要求各个成员必须要有很强的自我管理能力，同时具有一定的表达能力；成员可以采用任何工作方式，只要能达到Sprint的目标。

Scrum 开发流程

  上图种的 1-4week Sprint 为一个周期，也就是一次迭代。具体的流程如下：

  1. 首先需要确定一个Product Backlog（图左侧部分），由Product Owner 负责。
  2. 然后 Scrum Team会根据Product Backlog列表，做工作量的预估和安排。
  3. 有了Product Backlog列表后，需要通过Sprint Planning Meeting（Sprint计划会议），从中挑选出一个Story作为本次迭代完成的目标，这个目标的时间周期是1~4个星期，然后把这个Story进行细化，形成一个Sprint Backlog。
  4. Sprint Backlog是由Scrum Team去完成的，每个成员根据Sprint Backlog再细化成更小的任务。细到每个任务的工作量在2天内能完成。
  5. 在Scrum Team完成计划会议上选出的Sprint Backlog过程中，需要进行 Daily Scrum Meeting（每日站立会议），会议控制在15分钟左右，每个人都必须发言，并且要向所有成员当面汇报你昨天完成了什么，并且向所有成员承诺你今天要完成什么，同时遇到不能解决的问题也可以提出，每个人回答完成后，要走到黑板前更新自己的 Sprint burn down 图。
  6. 做到每日集成，也就是每天都要有一个可以成功编译、并且可以演示的版本；很多人可能还没有用过自动化的每日集成，其实TFS就有这个功能，它可以支持每次有成员进行签入操作的时候，在服务器上自动获取最新版本，然后在服务器中编译，如果通过则马上再执行单元测试代码，如果也全部通过，则将该版本发布，这时一次正式的签入操作才保存到TFS中，中间有任何失败，都会用邮件通知项目管理人员。
  7. 当一个Sprint Backlog被完成时，要进行 Srpint Review Meeting（演示会议），也称为评审会议，产品负责人和客户都要参加（最好本公司老板也参加），每一个Scrum Team的成员都要向他们演示自己完成的软件产品（非常重要）。
  8. 最后就是 Sprint Retrospective Meeting（回顾会议），也称为总结会议，以轮流发言方式进行，每个人都要发言，总结并讨论改进的地方，放入下一轮Sprint的产品需求中。

Scrum 的特征

  1. Transparency
  2. Time Boxing
  3. Communication
  4. Inspect & Adapt Model

eXtreme Programming

  规模上不同于scrum，xp开发主要是小团队。

eXtreme Programming的核心价值

  1. 沟通（Communication）
  2. 简单（Simplicity）
  3. 反馈（Feedback）
  4. 尊重（Respect）
  5. 勇气（Courage）

eXtreme Programming执行方法

  1. 团队工作
  2. 计划游戏
  3. 客户测试
  4. 简单设计
  5. 结对编程
  6. 测试驱动开发
  7. 改进设计
  8. 持续集成
  9. 集体代码所有权
  10. 编码标准
  11. 隐喻
  12. 可持续的速度
